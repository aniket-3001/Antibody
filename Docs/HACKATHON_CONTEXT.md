# Cognee Hackathon 2026

# Project Context

Version: 2.0

---

# Overview

This repository is being developed as a submission for the **Cognee Hackathon 2026**.

The objective is not simply to build an AI application.

The objective is to demonstrate the transformative value of **persistent AI memory** using Cognee's hybrid graph-vector memory engine.

Traditional LLMs are fundamentally stateless.

Every new conversation begins from scratch.

Once the context window is exceeded, valuable information disappears.

Cognee changes this paradigm by allowing AI systems to remember, recall, improve, and forget information across unlimited sessions.

MemoryOS exists to demonstrate why persistent memory fundamentally changes the way researchers interact with AI.

---

# Hackathon Theme

## Build AI That Never Forgets

The central challenge of the hackathon is:

> Build applications that become better because they remember.

Persistent memory should not be an optional feature.

Persistent memory should be the foundation of the application.

---

# Cognee

Cognee provides a persistent hybrid graph-vector memory layer.

Instead of repeatedly rebuilding context, applications continuously accumulate knowledge over time.

Cognee exposes four fundamental memory operations:

- remember()
- recall()
- improve()
- forget()

The project should naturally demonstrate all four.

---

# Cognee Memory Lifecycle

## remember()

Persist information into long-term memory.

Examples:

- Research papers
- PDFs
- URLs
- Markdown notes
- Experiment logs
- Benchmark results

---

## recall()

Retrieve information through:

- semantic similarity
- graph traversal

The application should emphasize relationship-aware reasoning rather than simple document retrieval.

---

## improve()

Memory should evolve.

Examples:

- new papers expand the graph
- experiments create additional relationships
- new knowledge enriches existing memory
- the research graph continuously grows

---

## forget()

Memory should be editable.

Users should be able to remove:

- papers
- experiments
- notes
- obsolete research

The graph should remain consistent after deletion.

Reasoning:

The Open Source Track aligns strongly with this project's strengths:

- software engineering
- AI systems
- graph memory

Although MemoryOS could later support Cognee Cloud, that is explicitly outside the scope of this hackathon MVP.

# Judging Criteria

Projects are evaluated using six dimensions.

---

# 1. Potential Impact

Does the project solve a meaningful problem?

Does persistent memory unlock something that is genuinely useful?

MemoryOS should clearly demonstrate why researchers benefit from long-term memory.

---

# 2. Creativity & Innovation

Avoid building another chatbot.

Avoid building another RAG application.

The project should demonstrate capabilities that become possible only because AI remembers across unlimited sessions.

---

# 3. Technical Excellence

Judges expect:

- clean architecture
- maintainable code
- modular design
- good engineering practices
- robust implementation

Cognee should be deeply integrated into the architecture.

---

# 4. Best Use of Cognee

This is arguably the most important judging criterion.

MemoryOS should naturally demonstrate:

- remember()
- recall()
- improve()
- forget()

Cognee should never feel like an optional storage layer.

Cognee should be the foundation of the application.

---

# 5. User Experience

The application should make persistent memory visible.

Users should immediately understand:

- what the system remembers
- how memory evolves
- why relationships matter
- why graph memory is valuable

Graph visualization should play a central role.

---

# 6. Presentation Quality

↓

Knowledge Graph

Persistent Recall

↓

Memory Evolution

# Success Strategy

The objective is **not** to build the largest application.

The objective is to maximize performance against the judging criteria.

Engineering decisions should prioritize:

1. Best demonstration of Cognee
2. Memorable live demonstration
3. Technical excellence
4. Clean architecture
5. Polished user experience

Feature count is intentionally a low priority.

---

# Development Philosophy

Do NOT build:

- another chatbot
- another PDF summarizer
- another RAG system
- another search engine

Instead build:

A persistent memory operating system.

Memory is the product.

The LLM is simply an interface to that memory.

---

# Scope Philosophy

This is a hackathon.

A polished MVP is more valuable than an unfinished ambitious platform.

Whenever scope increases, ask:

"Does this significantly improve the demonstration?"

If not, postpone it.

---

# Guiding Principle

Every feature should answer one question:

> Does this demonstrate the value of persistent memory?

If the answer is no, it should not be part of the MVP.

---

# Success Definition
